HyperText Transfer Protocol

  • 기반 프로토콜
    • TCP: HTTP/1.1, HTTP/2
    • UDP: HTTP/3
    • 현재는 HTTP/1.1을 주로 사용
  • 클라이언트 서버 구조
    • Request Response 구조
    • 클라이언트는 서버에 요청을 보내고, 응답을 대기
    • 서버가 요청에 대한 결과를 만들어서 응답
    • 비즈니스 로직, 데이터는 다 서버에 밀어넣는다.
    • 클라이언트는 UI, 사용성에 집중
    • 이렇게 나누어서 클라이언트와 서버는 독립적으로 진화할 수 있다.
  • 무상태 프로토콜(Stateless)
    • 문맥(context) 유지가 안됨: 중간에 다른 서버로 바뀌어도 된다.
      • 고객: 이 노트북 얼마인가요? 점원: 100만원 입니다. 고객: 노트북 2개 구매하겠습니다. 점원: 노트북 2개는 200만원 입니다. 신용카드, 현금중에 어떤 걸로 구매 하시겠어요? 고객: 노트북 2개를 신용카드로 구매하겠습니다. 점원: 200만원 결제 완료되었습니다.
    • 응답서버를 쉽게 바꿀 수 있다 → 무한한 서버 증설 가능(scale out 수평 확장 유리)
    • cf. stateful(상태유지): 중간에 다른 서버로 바뀌면 서버는 문맥을 모름
      • 고객: 이 노트북 얼마인가요? 점원: 100만원 입니다. 고객: 2개 구매하겠습니다. 점원: 무엇을 2개 구매 하시겠어요? 고객: 신용카드로 구매하겠습니다. 점원: 무슨 제품을 몇개 신용카드로 구매하시겠어요?
    • 실무 한계
      • 모든것을 무상태로 설계할 수 없다.
      • 로그인은 상태유지가 필요 → 브라우저 쿠키와 서버 세션등을 사용해서 상태 유지
      • 상태 유지는 최소한만 사용. 최대한 무상태로 설계한다.
      • 서버 개발자들이 어려워하는 업무… 같은시간에 딱 맞추어 발생하는 대용량 트래픽(티켓팅 등등..)
  • 비연결성
    • 연결을 유지하는 모델: 서버 자원을 계속 사용
    • 연결을 유지하지 않는 모델: 서버가 응답 보내고 tcp/ip 연결 종료
    • HTTP는 기본적으로 연결을 유지하지 않는 모델이다.(비연결성)
    • 일반적으로 초 단위 이하의 빠른 속도로 응답(1시간동안 수천명이 서비스 사용해도 실제 서버에서 동시에 처리하는 요청은 수십개 이하로 매우 작음)
    • 서버 자원을 매우 효율적으로 사용할 수 있다.
    • 단점/한계
      • TCP/IP 연결을 계속해서 새로 해야 함. → 3 way handshake 시간 추가
      • 웹 브라우저로 사이트를 요청하면 HTML 뿐 아니라 js, css, 추가 이미지 등 수많은 자원이 함께 다운로드됨
      • 현재는 HTTP 지속 연결(Persistent Connections)로 문제 해결
        • HTTP 초기 - 연결, 종료 낭비 연결 → 요청, HTML 응답 → 종료 연결 → 요청, js 응답 → 종료 연결 → 요청, 이미지 응답 → 종료
        • HTTP 지속 연결 연결 → 요청, HTML 응답 → 요청, js 응답 → 요청, 이미지 응답 → 종료
      • HTTP/2, HTTP/3에서 더 많은 최적화
  • HTTP 메시지
    • HTTP 메시지에 모든것을 전송: HTML, TEXT, IMAGE, JSON, XML…
    • 시작라인 start-line = request-line / status-line
      • 요청메시지(request-line)
        • method SP(공백) request-target SP HTTP-version CRLF(엔터)
          • method: GET, POST, PUT, DELETE… 서버가 수행해야 할 동작 지정
          • request-target: 절대경로 "/”로 시작하는 경로 absolute-path[?query]
      • 응답메시지(status-line)
        • HTTP-version SP status-code SP reason-phrase CRLF
          • status-code: HTTP 상태 코드. 요청 성공, 실패를 나타냄
          • reason-phrase: 이유 문구 OK, NOT FOUND 등등
    • HTTP 헤더 header-field = field-name “:” OWS(띄어쓰기 허용) field-value OWS
      • field-name
        • 요청메시지에서는 Host
        • 응답메시지에서는 Content-Type, Content-Length
      • 용도: HTTP 전송에 필요한 모든 부가정보
        • 메시지 바디의 내용, 크기, 압축, 인증, 요청 브라우저 정보… 등등
        • 필요시 임의의 헤더 추가 가능
  • 단순함, 확장 가능
    • HTTP는 단순하다. 스펙도 읽어볼만 함..
    • HTTP 메시지도 매우 단순
    • 크게 성공하는 표준 기술은 단순하지만 확장가능한 기술
  • 지금은 HTTP 시대!
